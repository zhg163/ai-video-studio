下面我按“**国产版 Flova / OpenCode + 视频能力**”的实际落地链路，给你一套**各阶段模型推荐 + 技术栈推荐 + 选型结论**。
我把推荐分成两条线：

* **闭源优先方案**：效果最好、上线最快
* **开源优先方案**：可私有化、成本可控、可二次开发

我先给结论，再给分阶段推荐。

---

# 一、总推荐结论

## 1）如果你的目标是“先把 MVP 做出来，尽快上线”

推荐走：

> **闭源主力 + 开源兜底** 的混合架构

最合适的组合是：

* **创意理解 / 脚本 / 分镜**：Qwen3.5-Plus / Qwen3-Max，备选 DeepSeek-V3.2
* **关键帧与角色设定图**：GPT Image 1 或 FLUX API；国产优先可接阿里百炼图像能力
* **视频生成主力**：Kling 3.0、阿里万相图生视频/首尾帧视频、MiniMax Hailuo 02
* **配音**：Qwen3-TTS（国产、可私有化优先），国际化可接 ElevenLabs
* **音乐**：MiniMax Music 或 ElevenLabs Music
* **剪辑合成**：FFmpeg + MoviePy/自研时间线服务，不建议这一层依赖大模型
* **工作流编排**：FastAPI + LangGraph + Redis + MongoDB + PostgreSQL + MinIO

原因是：
现在闭源视频模型在**可用性、成片质量、稳定性、推理速度**上仍然普遍强于开源栈；但开源模型非常适合做**私有化、降本、企业版兜底**。Kling 3.0 已经支持最长 15 秒、原生音视频输出与更灵活控制；阿里万相官方文档明确支持图生视频、首尾帧生视频，且 API 采用异步任务模式；MiniMax 也提供覆盖文本、语音、视频、图像、音乐的全栈模型矩阵。([Kling AI][1])

---

## 2）如果你的目标是“企业私有化 / 国产化 / 长期可控”

推荐走：

> **开源主力 + 闭源增强** 的双轨架构

最合适的组合是：

* **LLM**：Qwen3 / DeepSeek-V3.2
* **图像**：FLUX.1 [dev] / FLUX 系列
* **视频**：Wan2.1 / Wan2.1 VACE、HunyuanVideo / HunyuanVideo-I2V、CogVideoX
* **TTS**：Qwen3-TTS
* **编排**：FastAPI + LangGraph + Celery/Dramatiq + Redis Stack + MongoDB + PostgreSQL + MinIO
* **GPU 推理**：vLLM（文本），Diffusers / ComfyUI / 自研 worker（图像与视频）

Qwen 官方已发布 Qwen3 与 Qwen3.5-Omni/Qwen3-TTS 等系列；DeepSeek-V3 和 DeepSeek-V3.2 也已公开模型或 API；FLUX.1 [dev] 是开放权重图像模型；Wan2.1、HunyuanVideo、CogVideoX 都已有开源仓库或官方 Hugging Face / Diffusers 支持。([Qwen][2])

---

# 二、按生产阶段推荐模型

---

## 阶段 1：创意理解 / Brief 生成 / 脚本生成

这是整个系统的大脑层，要求：

* 中文能力强
* 结构化输出稳定
* 长上下文好
* 便于 Agent 调度
* 成本不能太高

### 闭源推荐

**首选：Qwen3.5-Plus / Qwen3-Max**
阿里云最新模型列表显示，Qwen3.5-Plus 支持文本、图像、视频输入，文本表现接近 Qwen3-Max，但更快、更具成本效益；对你的系统来说，它很适合做 brief 理解、脚本生成、镜头拆解、素材理解。([Alibaba Cloud][3])

**备选：DeepSeek-V3.2 API**
DeepSeek 官方在 2025 年底发布了 DeepSeek-V3.2 与 Speciale 版本，定位就是面向 agent 与推理增强场景；如果你更偏向“结构化生成 + 推理链 + 成本控制”，它很适合作为第二主力。([DeepSeek API Docs][4])

### 开源推荐

**首选：Qwen3**
Qwen 官方已发布 Qwen3 系列；做中文创意理解、剧情扩写、角色设定、脚本分段很合适。([Qwen][2])

**备选：DeepSeek-V3 / DeepSeek-V3.2-Exp**
DeepSeek-V3 官方仓库说明其为 671B 总参数、每 token 激活 37B 的 MoE 模型；同时 DeepSeek 也开放了 V3.2-Exp。你如果做私有化，DeepSeek 更适合放在“推理型 Agent”而不是极低延迟在线交互主链路。([GitHub][5])

### 我的建议

* **MVP 最优**：Qwen3.5-Plus 主用，DeepSeek-V3.2 作为复杂推理回退
* **私有化最优**：Qwen3 主用，DeepSeek-V3.2 做高复杂度离线生成

---

## 阶段 2：分镜生成 / Shot 拆解 / Prompt 展开

这一层本质还是 LLM，但比脚本更强调：

* JSON 输出稳定
* 镜头语言理解
* scene / shot / camera / action 结构化拆分
* 可强约束输出

### 推荐

这一层**不要单独换模型**，直接沿用阶段 1 的 LLM 即可：

* **闭源**：Qwen3.5-Plus / Qwen3-Max
* **开源**：Qwen3 / DeepSeek-V3.2

原因很简单：
分镜拆解不是“创作质量”问题，而是“结构化可靠性”问题。这里比拼的不是文学性，而是函数调用、JSON schema adherence、场景拆分能力。Qwen3.5-Plus 的多模态理解与成本/速度平衡更适合当主路由。([Alibaba Cloud][3])

### 我的建议

* 分镜层使用**单一主模型**，不要和脚本层频繁切换
* 做一个 **StoryboardAgent Prompt 模板中心**
* 输出统一 JSON，不要让前端直接吃自然语言

---

## 阶段 3：角色设定图 / 关键帧 / 分镜预览图

这一层目标是：

* 先把视觉世界“定住”
* 输出角色一致性参考图
* 给视频模型喂首帧/首尾帧
* 给用户看 storyboard 预览

### 闭源推荐

**首选：GPT Image 1**
OpenAI 官方文档显示 GPT Image 1 接受文本和图像输入，并输出图像，非常适合做图像生成与编辑。你如果希望“角色参考图 + 局部重绘 + 精修”一体化，GPT Image 1 很合适。([OpenAI Developers][6])

**备选：FLUX API（BFL）**
Black Forest Labs 官方提供 FLUX API，并明确可访问包括 Pro 层在内的模型；如果你更看重 photorealism 和工程接入灵活性，FLUX 也是很强的图像主力。([GitHub][7])

### 开源推荐

**首选：FLUX.1 [dev]**
FLUX.1 [dev] 是 12B 开放权重文本生图模型，适合私有化部署、角色设定图生成与参考图工作流。([Hugging Face][8])

**补充：FLUX Fill / Kontext 类能力**
如果你要做“参考图局部修改、服装替换、品牌物料植入”，FLUX Fill / Kontext 类图像编辑能力很实用。([Replicate][9])

### 我的建议

* **MVP**：GPT Image 1 或 FLUX API
* **企业私有化**：FLUX.1 [dev] + ComfyUI/自研 Diffusers worker
* 角色一致性不要只靠视频模型，先在图像层把角色 reference 固定住

---

## 阶段 4：视频生成（核心）

这是最关键的层。你要的不是学术 benchmark，而是：

* 画面质量
* 动作流畅性
* 镜头可控性
* 角色一致性
* API 稳定性
* 成本

### 闭源推荐

#### 方案 A：Kling 3.0 作为主力

Kling 官方文档和更新说明显示，Kling 3.0 系列支持最长 15 秒生成，支持原生音视频输出与更灵活的控制；2.6 也已支持单次生成中同时生成视觉、旁白、音效和环境氛围。对“成片感”和“镜头语言”要求高的产品，Kling 适合做主力商业视频生成。([Kling AI][1])

#### 方案 B：阿里万相作为国产主力

阿里云官方已经提供图生视频和首尾帧生视频 API，且明确是异步任务模式，任务通常需 1–5 分钟；这非常适合企业级后端工作流编排。对国产化、合规、内地部署更友好。([Alibaba Cloud Help Center][10])

#### 方案 C：MiniMax Hailuo 02 作为补充

MiniMax 官方模型发布页提到 Hailuo 02 支持 1080p、最长 10 秒视频生成；MiniMax 主页也明确其覆盖 text、speech、video、image、music 五大方向。它适合做第二视频供应商，防止单一模型故障或风格单一。([MiniMax API Docs][11])

#### 不建议作为 2026 新项目主力：Sora API

OpenAI 开发者文档明确写明 Sora 2 视频模型与 Videos API 已于 2026-03-24 宣布弃用，并将在 2026-09-24 下线。你可以研究它的提示工程，但不建议把新系统架构押在这条链路上。([OpenAI Developers][12])

### 开源推荐

#### 首选：Wan2.1 / Wan2.1 VACE

Wan 官方仓库显示已经发布 Wan2.1、VACE、FLF2V 等能力，覆盖视频生成与编辑，是当前最值得重点关注的开源视频主力之一。([GitHub][13])

#### 第二梯队：HunyuanVideo / HunyuanVideo-I2V

腾讯 HunyuanVideo 官方仓库与 Diffusers 文档都显示其目标是做到接近闭源视频基础模型的能力，HunyuanVideo-I2V 也已开放。适合做 image-to-video 与私有化研究。([GitHub][14])

#### 第三梯队：CogVideoX

CogVideoX 官方 GitHub 与 Diffusers 文档显示其支持更长、更一致的视频生成，且生态成熟，便于微调与工程接入。([GitHub][15])

### 我的建议

* **最快上线**：Kling 3.0 主力 + 阿里万相国产兜底 + MiniMax Hailuo 备用
* **私有化路线**：Wan2.1 主力 + HunyuanVideo / CogVideoX 补充
* **镜头策略**：MVP 阶段优先做 **image-to-video / 首尾帧视频**，不要把成片质量押在 text-to-video 一步到位上

---

## 阶段 5：配音 / 旁白 / 角色声音

这一层要求：

* 中文自然
* 情感表达
* 声线可控
* 支持克隆或 voice design
* 最好可流式

### 闭源推荐

**国际化首选：ElevenLabs**
ElevenLabs 官方文档说明其 TTS 支持富有情感的自然语音、多语言以及实时流式场景；官方 API 也覆盖 TTS、STT、Sound Effects、Music 等。([ElevenLabs][16])

### 开源/国产推荐

**首选：Qwen3-TTS**
Qwen 官方在 2026 年 1 月发布 Qwen3-TTS 家族，强调支持 voice clone、voice design、流式生成；其 Hugging Face 模型卡也直接链接到阿里云官方实时 API 文档。对你这种“国产版 Flova”项目，这是目前非常匹配的 TTS 主力。([Qwen][17])

### 我的建议

* **国产主链路**：Qwen3-TTS
* **国际化高拟真**：ElevenLabs
* 系统层做双提供商抽象，voice profile 不要和单一厂商绑死

---

## 阶段 6：音乐 / BGM / 音效

这一层不用追求“最好听”，要追求：

* 风格可控
* 时长可控
* 可自动 loop / 延展
* API 稳定

### 推荐

**MiniMax Music**
MiniMax 官方模型页已发布 Music 系列；其平台定位就是 full-stack multimodal。([MiniMax API Docs][11])

**ElevenLabs Music / Sound Effects**
ElevenLabs API 目前也覆盖 Music 和 Sound Effects。([ElevenLabs][18])

### 我的建议

* **MVP**：MiniMax Music 或 ElevenLabs Music 二选一即可
* 先解决“自动生成可用 BGM”，不要过早做音乐编辑器

---

## 阶段 7：剪辑合成 / 时间线 / 导出

这一层我不建议依赖大模型，最合适的是经典媒体栈：

* **FFmpeg**：最终拼接、转码、封面抽帧、字幕烧录
* **MoviePy / PyAV / 自研 timeline service**：做轻量时间线编辑
* **Whisper 类模型或厂商 ASR**：字幕对齐
* **Redis + 队列**：异步导出

### 我的建议

这一层的最佳技术不是“更强模型”，而是：

> **工程稳定性优先，媒体处理工具优先**

---

# 三、最合适的整体技术栈

结合你的目标，我给你一个“最合适，而不是最花哨”的推荐。

## 1）后端

**Python + FastAPI**

原因：

* 最适合对接模型生态
* 易接 LangGraph、vLLM、Diffusers、ComfyUI、FFmpeg
* 你本身也在用 FastAPI

---

## 2）Agent / 工作流

**LangGraph + 自研状态机**

原因：

* 你的系统本质是多阶段、可回滚、可重试的生产流水线
* LangGraph 适合 agent 状态流转
* 真正的任务控制仍建议你自己做 state machine 和 task orchestration

---

## 3）异步任务

**Celery 或 Dramatiq + Redis**

原因：

* 视频生成、音频生成、渲染都必须异步
* 任务重试、优先级、进度管理要成熟

---

## 4）数据层

**PostgreSQL + MongoDB + Redis Stack + MinIO**

这是你前面那套架构里最合适的组合：

* PostgreSQL：项目、任务、审计、权限
* MongoDB：script / storyboard / timeline 文档
* Redis Stack：进度、锁、缓存、幂等
* MinIO：图片、视频、音频、导出文件

---

## 5）前端

**Vue 3 + TypeScript**

原因：

* 你已有相关经验
* 适合做创作台、分镜编辑器、任务中心、时间线 UI

---

## 6）推理层

### 文本模型

* 托管：Qwen / DeepSeek API
* 私有：vLLM

### 图像与视频

* 闭源：统一接 API Gateway
* 开源：Diffusers / ComfyUI worker / 自研推理服务

### 音频

* 厂商 API + 自研封装层

---

# 五、我最推荐你现在怎么选

基于你一直在做的技术栈和产品路线，我的建议非常明确：

## 最优推荐

### 第一阶段先做这个组合

* **LLM**：Qwen3.5-Plus
* **图像**：FLUX API 或 GPT Image 1
* **视频**：Kling 3.0 + 阿里万相双路由
* **TTS**：Qwen3-TTS
* **音乐**：MiniMax Music
* **后端**：FastAPI + LangGraph + Redis + MongoDB + PostgreSQL + MinIO
* **渲染**：FFmpeg

这是当前最平衡的一条路：
既能快速上线，又保留国产化和私有化演进空间。Qwen3.5-Plus 在阿里模型列表中主打速度/成本平衡与更强多模态；Kling 3.0 和万相都已有成熟 API 形态；Qwen3-TTS 则非常适合你要做的中文视频创作链路。([Alibaba Cloud][3])

## 第二阶段再补

* 开源视频：Wan2.1 / HunyuanVideo
* 开源图像：FLUX.1 [dev]
* 高级推理：DeepSeek-V3.2
* 国际语音：ElevenLabs

这样做可以逐步把成本从闭源 API 拉回到可控区间，同时不影响第一版交付。Wan2.1、HunyuanVideo、CogVideoX 都已有公开实现或生态接入，适合你后续做企业版、自部署版。([GitHub][13])

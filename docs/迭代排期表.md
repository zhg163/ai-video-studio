# 迭代排期表

## 1. 文档目标

本文用于定义 `ai-video-studio` MVP 阶段的迭代排期、阶段目标、交付物、验收标准与依赖关系，作为研发执行与跟踪基线。

## 2. 排期原则

1. 先闭环，后增强。
2. 先后端主链路，后前端体验优化。
3. 先异步任务和版本体系，后做高级编辑能力。
4. 每个阶段必须具备可演示成果。
5. 每个阶段结束必须有明确验收口径。

## 3. 总体周期

MVP 计划周期：`10 周`

| 阶段 | 周期 | 核心目标 | 输出重点 |
|---|---|---|---|
| Phase 0 | 第 1 周 | 冻结方案与技术基线 | 设计、接口、数据模型 |
| Phase 1 | 第 2-3 周 | 打通项目到分镜链路 | Project/Brief/Script/Storyboard |
| Phase 2 | 第 4-6 周 | 打通素材生成与任务链路 | Image/Video/Audio/Task |
| Phase 3 | 第 7-8 周 | 打通时间线与导出闭环 | Timeline/Render |
| Phase 4 | 第 9-10 周 | 完成稳定化与上线准备 | Regenerate/监控/测试/部署 |

## 4. 详细排期

## 4.1 Phase 0：方案冻结

时间：`第 1 周`

目标：统一产品边界、服务拆分、接口风格、状态机和数据模型。

主要任务：

1. 冻结 MVP 范围与非目标范围。
2. 整理并统一现有 `docs` 文档中的设计结论。
3. 冻结核心领域模型与状态机。
4. 冻结 4 服务部署方案。
5. 冻结模型接入主策略与兜底策略。
6. 冻结 OpenAPI 风格与版本规范。

交付物：

1. `docs/详细设计文档` 或等价主设计稿
2. `docs/开发计划.md`
3. `OpenAPI` 基线文档
4. PostgreSQL DDL 草案
5. Mongo Schema 草案

验收标准：

1. 产品、后端、前端、测试对 MVP 范围无歧义。
2. 领域对象、状态机、接口命名风格统一。
3. 后续开发不新增核心对象类别。

依赖关系：

1. 无外部开发依赖。
2. 需要产品、架构、研发共同评审通过。

## 4.2 Phase 1：项目与创作基础链路

时间：`第 2-3 周`

目标：完成从项目创建到 Storyboard 生成的前半闭环。

主要任务：

1. 实现项目中心 API 与基础页面。
2. 实现 Brief 提交、结构化、查看、编辑。
3. 实现 Script 生成、查看、编辑、版本化。
4. 实现 Storyboard 生成、Shot 编辑、时长编辑、Prompt 编辑。
5. 落地 `CreativeAgent`、`ScriptAgent`、`StoryboardAgent`。
6. 建立 PostgreSQL + MongoDB 双存储基础。
7. 建立项目状态流转基础逻辑。

交付物：

1. 项目中心页面
2. Brief 页面
3. Script 页面
4. Storyboard 页面
5. 项目、Brief、Script、Storyboard API

验收标准：

1. 用户可以创建项目并提交创意。
2. 用户可以生成并编辑 Brief。
3. 用户可以生成脚本并查看版本。
4. 用户可以生成分镜并编辑 Shot。

依赖关系：

1. Phase 0 设计基线完成。
2. 基础数据库和对象存储可用。

## 4.3 Phase 2：媒体生成与任务系统

时间：`第 4-6 周`

目标：完成单镜头素材生成链路和统一任务系统。

主要任务：

1. 实现 Image Gateway 与关键帧生成流程。
2. 实现 Video Gateway 与单镜头视频生成流程。
3. 实现 TTS、BGM、字幕基础接口。
4. 实现 `VisualAgent` 与 `AudioAgent`。
5. 建立 `generation_task` 数据结构与任务中心。
6. 实现异步队列、轮询、进度、取消、重试、幂等。
7. 实现候选素材列表与选中逻辑。
8. 建立资产上传和引用关系。

交付物：

1. Shot 图像生成能力
2. Shot 视频生成能力
3. 配音、字幕、BGM 基础能力
4. 任务中心页面
5. 素材管理基础能力

验收标准：

1. 单个 Shot 可以生成关键帧和视频。
2. 用户可以查看候选素材并选择结果。
3. 用户可以查看任务状态、进度和错误原因。
4. 音频与字幕可以成功生成并存储。

依赖关系：

1. Storyboard 与 Shot 结构已稳定。
2. 模型提供商接口和密钥准备完成。
3. Redis 队列、MinIO 存储可用。

## 4.4 Phase 3：时间线与导出闭环

时间：`第 7-8 周`

目标：实现从素材到成片的装配与导出闭环。

主要任务：

1. 实现 Timeline 数据结构与版本化。
2. 实现自动装配逻辑。
3. 实现视频轨、配音轨、BGM 轨、字幕轨基础能力。
4. 实现片段重排、替换、裁剪、音量调整。
5. 实现 `EditingAgent`。
6. 实现 Render 流程、FFmpeg 合成、导出记录。
7. 实现预览与下载功能。

交付物：

1. Timeline API 与页面
2. Render API 与导出页
3. 导出任务追踪页
4. 可下载成片

验收标准：

1. 用户可以自动生成 Timeline。
2. 用户可以发起导出并查看导出状态。
3. 导出结果可预览、可下载。
4. RenderTask 与 TimelineVersion 绑定正确。

依赖关系：

1. Shot 视频、音频、字幕资产已可用。
2. FFmpeg 运行环境已部署完成。

## 4.5 Phase 4：稳定化与上线准备

时间：`第 9-10 周`

目标：补齐重生成、监控、测试和部署能力，完成 MVP 上线准备。

主要任务：

1. 实现 Shot 级重生成。
2. 实现素材替换与回退逻辑。
3. 完善失败补偿、重试、取消策略。
4. 增加审计日志、模型调用日志、成本日志。
5. 接入 Prometheus、Grafana、Loki 基础监控。
6. 补齐接口测试、集成测试、主链路验收测试。
7. 完成 Docker / 预发部署脚本。
8. 输出上线检查清单。

交付物：

1. Shot 重生成能力
2. 基础监控面板
3. 测试报告
4. 部署文档
5. 上线验收报告

验收标准：

1. 主链路成功率达到预期。
2. 常见失败可追踪、可重试、可恢复。
3. 具备最小可运维能力。
4. 上线演示流程完整可跑通。

依赖关系：

1. 前三阶段已完成。
2. 预发环境和日志系统就绪。

## 5. 跨阶段持续事项

以下事项贯穿全周期：

1. API 文档维护
2. DDL / Schema 同步
3. 代码规范与评审
4. 日志规范统一
5. 安全检查
6. 成本监控
7. Demo 数据和演示脚本维护

## 6. 关键检查点

### 第 1 周检查点

1. 设计评审完成。
2. 接口风格冻结。
3. 数据模型冻结。

### 第 3 周检查点

1. 从创意到 Storyboard 可跑通。
2. Script 与 Storyboard 可版本化。

### 第 6 周检查点

1. 单 Shot 图像与视频生成可用。
2. 任务中心可用。
3. 音频和字幕基础能力可用。

### 第 8 周检查点

1. Timeline 可自动装配。
2. 导出 MP4 可成功。

### 第 10 周检查点

1. Shot 重生成可用。
2. 监控和日志可用。
3. MVP 验收通过。

## 7. 资源投入建议

建议最小团队配置：

1. 产品/架构：1 人
2. 后端：2 人
3. 前端：1-2 人
4. 测试：1 人
5. 运维/平台：0.5-1 人

如果资源不足，优先级顺序为：

1. 后端主链路
2. 前端创作基础页
3. 任务中心与导出
4. 管理后台

## 8. 风险节点

1. 第 4-6 周是最大风险阶段，模型生成链路和异步任务系统最容易拖期。
2. 第 7-8 周是第二风险阶段，Timeline 和 FFmpeg 导出链路集成复杂度较高。
3. 第 9-10 周如果前面链路不稳定，稳定化周期会被压缩。

## 9. 排期结论

MVP 的关键不是平均推进所有模块，而是确保每个阶段都服务于最终闭环：

1. 前两阶段完成“创意到分镜”
2. 中间阶段完成“分镜到素材”
3. 后两阶段完成“素材到成片”

只要严格按照闭环主线推进，并控制非核心需求扩散，`10 周` 内完成 MVP 是可执行的。
